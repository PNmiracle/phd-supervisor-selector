# 管理员操作规范

> 本文档给**仓库管理员（PNmiracle）**看。
> 目标：规范合并流程，保证仓库历史干净，协作顺畅。

---

## 角色说明

| 角色 | 权限 | 职责 |
|------|------|------|
| **管理员（你）** | Maintainer | Review PR、合并、解决复杂冲突、更新 SKILL.md |
| **协作者（同事）** | Collaborator | 改文件、开 PR、不直接 push main |

---

## 日常操作流程

### 收到 PR 通知后

**第1步：打开 PR 页面**

GitHub 会发邮件通知，或者直接在：
```
https://github.com/PNmiracle/phd-supervisor-selector/pulls
```

**第2步：Review 改动**

点 **"Files changed"** 标签，看协作者改了哪些行：
- 绿色 = 新增
- 红色 = 删除

**检查要点：**
- [ ] 改动是否符合预期（比如只改了学校策略，没动 SKILL.md 核心规则）
- [ ] 有没有语法错误（比如 Markdown 格式错了）
- [ ] 有没有把敏感信息提交上来（比如 API Token）

**第3步：提出修改意见（可选）**

如果发现问题，点对应行的 `+` 号，写评论：

```
建议把"NUS"改成"新加坡国立大学"，备注里也统一一下～
```

然后点 **"Start a review"** → **"Request changes"**

协作者会收到通知，修改后重新推送，PR 会自动更新。

**第4步：合并 PR**

如果没问题，点页面底部的：

**"Merge pull request"** → **"Confirm merge"**

**合并方式选择：**

| 方式 | 什么时候用 |
|------|------------|
| **Merge commit**（默认） | 日常用这个，保留完整历史 |
| **Squash and merge** | 协作者提交了很多琐碎的 commit（比如改了又改），合并成一个 |
| **Rebase and merge** | 你很确定没冲突，想保持线性历史 |

**推荐：日常用 "Merge commit" 就行。**

**第5步：删除分支（可选）**

合并后 GitHub 会提示 **"Delete branch"**，点一下保持仓库整洁。

## 处理冲突 PR

如果 PR 页面显示 **"This branch has conflicts that must be resolved"**：

**方式A：让协作者解决（推荐）**

在 PR 页面写评论：
```
@协作者GitHub用户名 这个 PR 和 main 有冲突，麻烦你本地说"同步技能"重新提交一下～
```

协作者说"同步技能"时，AI 会自动处理冲突。

**方式B：管理员直接解决（紧急情况）**

1. 点 PR 页面的 **"Resolve conflicts"**
2. GitHub 会显示冲突内容（带 `<<<<<<<` 标记）
3. 手动编辑，删掉 `<<<<<<<`、`=======`、`>>>>>>>` 这些标记
4. 点 **"Mark as resolved"** → **"Commit merge"**

## 维护 SKILL.md

**只有管理员能改 SKILL.md 的核心规则部分。**

协作者可以改 `references/*.md`，但不应该改 `SKILL.md` 里的规则。

**如果你改了 SKILL.md：**

1. 改完，提交并 push 到 main
2. **通知协作者**："我更新了 SKILL.md，下次说'同步技能'前，先让 AI 执行 `git pull`"

---

## 版本号管理（可选）

如果 SKILL.md 有大的改动，可以在文件顶部加版本号：

```markdown
---
name: phd-supervisor-selector
version: 1.2.0
updated: 2026-07-04
---
```

这样协作者能看清自己本地是不是旧版本。

## 协作规范

### ✅ 鼓励的做法

- 协作者开 PR 前，先在微信群里说一声"我改了XX，准备提交"
- 管理员收到 PR，尽量当天内 review 并合并
- 大的改动（比如加新功能、改核心规则），先在微信群里讨论再动手

### ❌ 禁止的做法

- **协作者不要直接 push main**（应该通过 PR）
- **管理员不要在没有 review 的情况下直接合并自己的 PR**（虽然你是管理员，但最好也让同事看一眼）
- **不要在 commit 消息里写敏感信息**（比如 "fix: 把 token 改成了 uskXXX"）

### 每周一次：清理已合并的分支

**对 AI 说：**
> "帮我看看 GitHub 上有哪些已经合并但还没删的分支，清理一下"

---

### 每月一次：检查 SKILL.md 是否和实际操作一致

用了一阵子后，可能会发现 SKILL.md 里写的流程和实际操作不一样。

**这时候：**
1. 更新 SKILL.md
2. 提交并 push
3. 通知协作者 "SKILL.md 更新了，下次注意看变化"

## 紧急情况处理

### 情况1：协作者不小心把敏感信息 push 上去了

**立即行动：**

1. 在 PR 页面点 **"Close pull request"**（如果还没合并）
2. 如果已经合并了，点 **"Revert"** 撤销这次提交
3. 让协作者 **换一个新 Token**，旧的视为已泄露

---

### 情况2：有人恶意提交（虽然不太可能，但防着点）

1. 点 PR 页面 → **"Close pull request"**
2. 在仓库 Settings → **"Collaborators"** 里移除这个人
3. 检查提交历史，看有没有被篡改

---

## 给协作者的反馈模板

**LGTM（Looks Good To Me，可以合并）：**
```
LGTM ～ 改得挺好的，已合并 ✅
```

**需要修改：**
```
有几个小问题：
1. XX 文件的 YY 行，建议改成 ZZ
2. Commit 消息可以写得更清楚一点～

改完重新说"同步技能"就行哈～
```

**询问意图：**
```
你这次改 XX 是想实现什么功能呀？我看看有没有更好的实现方式～
```

## 总结：你需要记住的命令

| 场景 | 操作 |
|------|------|
| 收到 PR 通知 | 打开 PR 页面 → Review → Merge |
| PR 有冲突 | 让协作者重新"同步技能"，或自己点 "Resolve conflicts" |
| 想看所有 PR | 打开 `https://github.com/PNmiracle/phd-supervisor-selector/pulls` |
| SKILL.md 有更新 | 提交并 push → 通知协作者 |

**就这些，不需要记更多。**

## 附录：gh CLI 常用命令（可选）

如果你喜欢用命令行操作 PR：

```bash
# 列出所有开放的 PR
gh pr list

# 查看某个 PR 的详情
gh pr view 12

# 合并某个 PR（需要 PR 号）
gh pr merge 12 --merge

# 检查 PR 状态
gh pr checks 12
```

**但推荐用网页版**，因为能看 diff，更直观。
