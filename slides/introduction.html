01 / 08

每个大学网站
都是一扇不同的门

不是"搜个关键词就行"——得想办法敲开每扇门

🏰

没有统一入口

100 所大学，100 种网站系统。WordPress、Pure 平台、自研系统——每家都不一样。

🛡️

各种"门禁"

Cloudflare 防火墙、JS 动态加载、反爬虫检测——戒备比想象中森严得多。

📋

不能只看不验

找到名字 ≠ 找到导师。必须验证页面、确认方向、判断是否带博——每一步都要核对。

💡
打个比方：给 100 个不同小区送快递
每个小区门禁都不一样
刷脸、密码、按门铃等保安——没有万能钥匙

03 / 08

实际用的五招
从温柔到暴力

根据每个网站的性格，选择不同策略

🚪 第一招

直接敲门 · curl 请求

伪装浏览器身份直接访问 staff 页面，解析 HTML 提取人名链接。对老老实实用服务器渲染的站最有效。

🗝️ 第二招

找后门 · JS API 挖掘

下载 JS 文件，搜索 API 地址，找到真正的数据接口——绕过壳子直接拿 JSON。

📖 第三招

搜索引擎按名索骥

拿到导师名字+学校 → 直接搜索引擎搜 "[名字] [学校] professor" → 获取真实存活 URL → curl + 浏览器验证。

🤖 第四招

智能分配·多个子智能体并行处理

同时分配多个 AI 子智能体分头搜索不同学校，最后汇总——速度翻倍。

04 / 08

不是搜到就停
而是越挖越深

Deep Discovery Pipeline —— 每个学校都挖到不能再挖为止

🔁

按需回查

不在每轮搜索中自动重试 404——只在用户明确要求或有关键信号（新学年、网站改版）时才回查，避免浪费 token。

🌿

跨院系联想

不只搜"服装系"，还搜人类学、文化研究、设计史——导师可能藏在隔壁院系。

🏷️

隐藏标签点击

自动点击页面隐藏 tab（Research Interest、Publications）——关键信息常藏在折叠面板后。

🧪

备用域名探测

主站打不开？试试研究生院、研究门户子域名——同一大学不同系统防护等级不同。

📊

API 批量拉取

找到 API 后一次拉全部数据到本地筛——不翻页不逐页爬，效率高几十倍。

05 / 08

遇到盾牌怎么办？
不放弃，也不死磕

Cloudflare 反爬、JS 渲染壳、403 拦截……每种都有对应策略

🛡️ Cloudflare

换路径

主站被拦 → 试试 Pure 研究门户、研究生院、ePrints。同校不同系统防护不同。

💻 JS 渲染壳

找 API

下载 JS 搜 api、endpoint、baseUrl——找到真正的数据接口，绕过壳子拿 JSON。

🚫 全部失败

标记⚠️不删

彻底打不开的保留在表中标注 "⚠️需手动验证"——不让任何候选人遗漏。

🏋️
核心韧性：搜索引擎优先找 URL → curl 验证 → 浏览器模拟人确认内容 → JS 渲染站挖 API → 全失败标记 ⚠️。不删任何候选人，学生知道哪些需要手动确认。
06 / 08

和"关键词搜索"
有什么不同？

三列对比，一目了然

🔤 普通关键词搜索
  • ✗ 只返回有 SEO 的页面
  • ✗ 无法穿透 JS 渲染站
  • ✗ 不验证页面真伪
  • ✗ 不判断是否带博
  • ✗ 重复结果多、遗漏多
🦾 phd-supervisor-selector
  • ✓ 搜索引擎搜名+校名找真实 URL
  • ✓ 挖 JS API + 浏览器验证
  • ✓ 逐页验证姓名/职称/方向
  • ✓ 人工判断是否可带博
  • ✓ 模拟人操作浏览器验证页面
  • ✓ 去重 + 排除 + 备注 + 分类
🕸️ 传统爬虫
  • ✓ 能批量抓取
  • ✗ 遇到 Cloudflare 就死
  • ✗ 不会自己找 API
  • ✗ 不理解研究方向含义
  • ✗ 不会判断匹配度
🎯
本质区别：关键词搜索 = 在商场门口发传单。phd-supervisor-selector = 拿着学生简历，去每个院系办公室敲门问"你们这儿有没有合适带博士的老师？"——有目标、有判断、有验证。
07 / 08

不止是搜
还要品控

搜到的候选人要经过多道关卡才能进入正式推荐表

✅

主页必须能打开

不是 200 状态码就行——页面内容必须包含导师名字、职称、院系信息。空的 JS 壳不算。

🔗

三个链接都要有

导师主页 + 博士申请页 + 院系教员列表——每个候选人都配齐三个独立验证过的链接。

📝

备注只写事实

不写"完美匹配""高度推荐"——只写客观信息：职称、研究关键词、风险提示。让学生自己判断。

🗑️

不合格的进排除表

教学岗、退休、方向不匹配、页面打不开——不直接删除，而是移到"排除或待确认"表，理由写清楚。

🎨

每轮搜索标颜色

新增的导师用浅色高亮标记，说明表里解释每种颜色代表哪一轮搜索——清楚追溯。

08 / 11

这套流程
还能怎么优化？

量大管饱 · 匹配准确 · 链接可验证

🔄

每次重新探路

搜同一个学校，每次都要重新试 URL、找 API、猜路径——浪费时间。

🐢

子代理不够多

只用 1 个子代理串行搜索，等它跑完才能看结果——速度太慢。

🔒

只盯大学官网

官网被 Cloudflare 挡了就束手无策——其实还有 ORCID、Scholar 等渠道。

👀

匹配靠人眼

搜回来手动看研究方向是否匹配——量大时很吃力。

🔗

链接逐个验

交付前不批量验证——可能漏掉死链。

09 / 11

优先改进 P0
成本最低 · 收益最大

立即可做，马上见效

🗂️

大学档案库

给每个常见大学建一张"策略卡"：记住它的网站类型、最佳访问方式、已知 API 端点。下次搜同一学校不用重新踩坑——直接走已知路径。

🚀

子代理并行 × 10

每个学校分配一个独立子智能体，5-10 个同时跑。互不干扰，主代理只负责汇总——速度翻 5-10 倍。

⚡
以前：一个人拿着一张地图，挨家挨户跑。
以后：同时派出 10 个人，每人拿着一张写了"这家的门怎么开"的攻略卡。
10 / 11

后续改进 P1
新渠道 · 新能力

需要一些开发，但提升显著

🌐

ORCID 公开 API

学者自己维护的研究方向 + 机构信息，结构化 JSON 数据——绕过大学防火墙，从学者侧发现候选人。

📚

Semantic Scholar

免费学术 API，按研究领域检索学者——覆盖面广，不受学校反爬影响。

🔗

批量链接健康检查

搜完后对所有候选人主页做一次 HEAD 请求，一次性标出死链——交付前确保可打开。

🎯

自动匹配度打分

用学生方向关键词跟导师研究描述做相似度计算——量大时自动筛，人工只看高分候选。

🧠

跨院系智能推荐

不只搜本专业——自动联想相关院系（人类学→物质文化、文化研究→批判时尚），拓宽候选人池。
